**Autonomous multi-agent AI system that detects, analyzes, and responds to cybersecurity threats in real time.**

Six specialized AI agents work in a sequential pipeline — from log ingestion and anomaly detection through threat intelligence enrichment, forensic correlation, risk scoring, automated containment, to executive-level incident reporting.

Includes **VulnLab** — a bundled intentionally vulnerable web application that serves as a live attack surface for real-time threat detection testing.

## 🚀 Key Features

### 🤖 Multi-Agent AI Investigation Pipeline
- **6 autonomous AI agents** (Sentry, Hunter, Detective, Judge, Medic, Scribe) orchestrated by CrewAI in a sequential pipeline
- Each agent has a specialized role, dedicated tools, and produces structured output consumed by the next agent
- Supports **Groq Cloud** (LLaMA 3.3 70B) and **Ollama** (DeepSeek-R1) as LLM backends — switchable via environment variable

### 📡 Real-Time Threat Monitoring
- **Multi-source log watcher** that tails `/var/log/auth.log`, UFW, nginx, audit.log, DHCP, DNS, and syslog simultaneously
- **Windows Event Log** support (Event IDs 4624/4625/4648 via pywin32)
- **VulnLab Integration** featuring a dedicated live-tailer for `vulnlab_access.log` (Nginx format) generated by the bundled vulnerable application on port 9090
- **Synthetic log generation** fallback when no real log sources are available
- **Server-Sent Events (SSE)** for zero-latency live streaming to the frontend
- **GeoIP enrichment** via ip-api.com with in-memory caching

### 🧠 Machine Learning Anomaly Detection
- **IsolationForest-based** anomaly scoring on per-IP behavioral features
- Rolling 5-minute window analysis of failed login counts, unique endpoint diversity, error rates, and request velocity
- Anomalous IPs are flagged and injected into the Sentry agent's context before crew execution

### ⚙️ Configurable Threshold Engine
- **11 configurable thresholds** covering SSH brute force, firewall blocks, port scanning, SQL injection, HTTP errors, rogue devices, DNS anomalies, and rate limiting
- Sliding-window evaluation on every incoming log row
- Alerts persisted to JSON with configurable severity mapping
- **Hot-reloadable** thresholds via API — no restart required

### 🚨 Automated Incident Response
- **Auto-Response Engine** generates and optionally executes firewall containment commands
- **Windows:** `netsh advfirewall` rules | **Linux:** `iptables` / `ufw` rules
- **Blocked IP Management** — persistent `blocked_ips.json`, REST API for block/unblock, and Live Monitor UI panel
- Dry-run mode by default — logs commands without executing for safety
- Full audit trail persisted to `response_audit.json`

### 📧 Email Alerting (SMTP)
- Gmail SMTP integration with TLS/SSL support
- **Dynamic per-user alerting** — emails sent to currently logged-in users' addresses
- Per-IP cooldown tracking to prevent alert fatigue
- Configurable minimum severity threshold
- **Thread-safe fire-and-forget** email dispatch from background workers
- HTML-formatted alert emails with dashboard links

### 🔍 Port Scanning & Exposure Analysis
- **Nmap integration** for service version detection with XML parsing
- Allowlist-based unexpected port flagging
- Periodic automated scans (every 30 minutes)
- Mock fallback when nmap is not installed
- Scan history persistence (last 50 scans)

### 🔴 Redis Pub/Sub Pipeline
- Optional Redis integration for cross-process log distribution
- Pub/Sub channel + Redis Streams for durability (capped at 50K entries)
- Graceful degradation to in-memory queues when Redis is unavailable
- One-command setup via Docker Compose

### 🔒 Authentication & Authorization
- **JWT-based authentication** with bcrypt password hashing
- **Google OAuth 2.0** one-click sign-in
- MongoDB-backed session management with expiry enforcement
- Protected API routes with automatic token refresh on the frontend

### 📂 Incident History & Persistence
- All incidents, reports, and logs persisted to **MongoDB** (5 collections: users, sessions, incidents, logs, reports)
- Full CRUD API for incident management with filtering
- Structured report parsing from AI-generated Markdown into queryable JSON

### 🎯 VulnLab — Integrated Attack Surface
- **Deliberately vulnerable Flask web application** bundled for live testing
- Exposes 10+ vulnerabilities: SQL Injection, XSS, IDOR, Command Injection, Unrestricted File Upload, Broken Access Control, and more
- Generates **nginx-format access logs** consumed by Cyberguard in real time
- Includes a **DDoS attack simulator** for stress-testing detection capabilities
- Pre-seeded SQLite database with test users and data

### 🖥️ Rich Frontend Dashboard
- Real-time dashboard with stat cards, live agent feed, anomaly detection panel, and incident summary
- **Threat Monitor** page with live SSE feed table, stats bar, threshold configuration panel, alert history timeline, and port exposure scanner
- **Report page** with styled Markdown rendering, structured header card, and PDF export
- **Investigation page** showing the 6-agent pipeline workflow and detailed agent descriptions
- **History page** with incident cards, expandable details, severity/attack-type filtering, and delete functionality
- **About page** with project overview, team info, and technology breakdown
- Excel/CSV export for alert history via SheetJS

## ⚡ Quick Start

### 1. Clone the Repository

```bash
git clone https://github.com/himanshubijja/KLH_hackathon.git
cd KLH_hackathon
```

### 2. Backend Setup

```bash
cd backend

# Install dependencies using uv (recommended)
uv sync

# Or using pip:
python -m venv .venv
# Windows: .venv\Scripts\activate
# Linux/Mac: source .venv/bin/activate
pip install -e .
```

### 3. Configure Environment Variables

Edit `backend/.env` with your API keys:

```env
# Required: LLM Backend
GROQ_API_KEY=your_groq_api_key_here
MODEL=groq/llama-3.3-70b-versatile

# Or use local Ollama instead:
# LLM_BACKEND=ollama
# OLLAMA_MODEL=deepseek-r1:1.5b

# Required: MongoDB
MONGODB_URI=mongodb://localhost:27017
MONGODB_DB=cyberguard_db

# Required: JWT Secret
JWT_SECRET_KEY=your_random_secret_key_here

# Threat Intelligence APIs
ABUSEIPDB_API_KEY=your_abuseipdb_key_here
VIRUSTOTAL_API_KEY=your_virustotal_key_here

# Optional: Google OAuth
GOOGLE_CLIENT_ID=your_google_client_id_here

# Optional: Email Alerting
SMTP_ENABLED=false
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=your@gmail.com
SMTP_PASSWORD=xxxx_xxxx_xxxx_xxxx
ALERT_EMAIL_TO=recipient@gmail.com

# Optional: Redis
# REDIS_URL=redis://localhost:6379/0
```

Edit `frontend/.env`:

```env
VITE_API_BASE_URL=http://localhost:8000
VITE_GOOGLE_CLIENT_ID=your_google_client_id_here
```

### 4. Start MongoDB

```bash
# Using Docker:
docker run -d -p 27017:27017 --name cyberguard-mongo mongo:7

# Or install locally: https://www.mongodb.com/docs/manual/installation/
```

### 5. Start Redis (Optional)

```bash
# Using Docker Compose (from project root):
docker compose up -d

# This starts Redis 7 on port 6379 with AOF persistence
```

### 6. Start the Backend

```bash
cd backend
uv run uvicorn server:app --reload --host 0.0.0.0 --port 8000
```

### 7. Frontend Setup

```bash
cd frontend
npm install
npm run dev
```

### 8. Start VulnLab (Optional — for live attack testing)

```bash
cd test_app

# Initialize the database (first run or to reset)
uv run python init_db.py

# Start the vulnerable application
uv run python app.py
```

VulnLab is now running at **http://localhost:9090** and writing access logs to `backend/data/vulnlab_access.log`.

### 9. Open the Application

Navigate to **http://localhost:5173** in your browser.

1. **Sign up** for a new account (or use Google OAuth)
2. **Upload a CSV** log file or click **Run Simulation** to use generated logs
3. Watch the **6 AI agents** analyze threats in real time
4. View the **Incident Report**, **Threat Monitor**, and **History** pages
5. Optionally, attack VulnLab and watch Cyberguard detect threats live

## ⚔️ Attack Simulation

CyberGuard includes **VulnLab** — a deliberately insecure Flask app for real-time security testing. Attack it from your browser, phone, or another device on the same network, and watch CyberGuard detect and respond.

### From a browser (phone or another device):

```
http://<YOUR_IP>:9090/login                           # Brute force target
http://<YOUR_IP>:9090/products?search=' OR 1=1--      # SQL Injection
http://<YOUR_IP>:9090/ping                            # Command Injection
http://<YOUR_IP>:9090/ddos                            # DDoS simulator UI
```

### From terminal (`ddos_simulate.py`):

```bash
cd test_app

# HTTP flood (30s, 20 threads)
python ddos_simulate.py --target <YOUR_IP> --port 9090 --attack http --duration 30 --threads 20

# Mixed attack (all 3 attack types, 60s, 50 threads)
python ddos_simulate.py --target <YOUR_IP> --port 9090 --attack mixed --duration 60 --threads 50

# Slowloris connection starvation
python ddos_simulate.py --target <YOUR_IP> --port 9090 --attack slowloris --duration 60
```

### Attack types detected by CyberGuard:

| Attack | Status | Severity | Response |
|---|---|---|---|
| **Brute Force** (failed logins) | `Failed_Login` | HIGH | Email alert + auto-block |
| **SQL Injection** | `SQL_Injection` | CRITICAL | Immediate email + auto-block |
| **DDoS / HTTP Flood** | Burst detection | CRITICAL | Rate-limit + auto-block |
| **Directory Scanning** | `404_Not_Found` | MEDIUM | Alert on threshold breach |
| **Command Injection** | `Command_Injection` | HIGH | Email alert |
| **Port Scanning** | `Port_Scan` | HIGH | Alert on unique ports threshold |

## 🎯 VulnLab — Integrated Attack Surface

> ⚠️ **VulnLab is INTENTIONALLY INSECURE. It is designed for local security testing only — never deploy it to a public network.**

VulnLab is a deliberately vulnerable Flask web application included in the `test_app/` directory. It serves as a **live attack surface** that Cyberguard monitors in real time.

### How It Works

```
VulnLab (port 9090) → nginx-format access logs → backend/data/vulnlab_access.log
                                                        ↓
                                              log_watcher.py (real-time tailing)
                                                        ↓
                                              Cyberguard AI Agents → Alerts & Reports
```

### Exposed Vulnerabilities

| Vulnerability | Route | Description |
|---|---|---|
| **SQL Injection** | `/login`, `/products`, `/api/users` | Raw f-string SQL queries with no parameterization |
| **Stored XSS** | `/comments` | User comments rendered as raw unescaped HTML |
| **Command Injection** | `/ping`, `/logs` | User input passed directly to `subprocess.run(shell=True)` |
| **IDOR** | `/profile/<user_id>` | Access any user's profile without authorization |
| **Unrestricted File Upload** | `/upload` | No file-type validation, no size limits |
| **Broken Access Control** | `/admin` | Admin panel accessible without authentication |
| **Info Disclosure** | `/api/system-info`, `/config` | Exposes hostname, OS, DB path, secret key |
| **Sensitive Data Exposure** | `/.env`, `/backup/*`, `/secret/*` | Credentials and backups served directly |

## 🤖 AI Agents & CrewAI Pipeline

The core of Cyberguard is a **6-agent sequential pipeline** powered by CrewAI. Each agent has a specialized role, custom tools, and produces output that feeds into the next agent.

### Agent Pipeline

```
CSV Logs → SENTRY → HUNTER → DETECTIVE → JUDGE → MEDIC → SCRIBE → Incident Report
```

| # | Agent | Role | Tools | What It Does |
|---|---|---|---|---|
| 1 | **🔍 SENTRY** | Log Analyst | `analyze_log_anomalies` | Scans logs for brute force, SQL injection, and port scanning patterns. Produces a list of suspicious IPs with attack classifications. |
| 2 | **🌐 HUNTER** | Threat Intelligence | `check_threat_intelligence` | Queries AbuseIPDB, VirusTotal, and Shodan InternetDB for each suspicious IP. Returns reputation scores, malware flags, and open ports. |
| 3 | **🕵️ DETECTIVE** | Forensic Correlator | `build_attack_timeline` | Reconstructs a chronological timeline of suspicious events. Correlates attack phases and identifies the attack narrative. |
| 4 | **⚖️ JUDGE** | Risk Scorer | None (LLM reasoning) | Calculates a quantitative risk score (0-100) using a strict weighted rubric (SQLi +50, Brute Force +30, etc.) with sanity checks mapping to severity labels (CRITICAL/HIGH/MEDIUM/LOW). |
| 5 | **🩺 MEDIC** | Incident Responder | `execute_containment_playbook` | Generates `iptables` firewall commands and user lockout actions for each malicious IP. Simulates containment execution. |
| 6 | **📝 SCRIBE** | Report Writer | None (LLM generation) | Produces a board-level incident report in Markdown with executive summary, Cyber Kill Chain analysis, IOCs, risk quantification, and remediation steps. |

### Custom CrewAI Tools

| Tool | Description |
|---|---|
| `Read Security Logs` | Parses and formats CSV log data for agent consumption |
| `Analyze Log Anomalies` | Detects brute force, SQL injection, and port scanning patterns with severity ratings |
| `Check Threat Intelligence` | Multi-source threat fusion: AbuseIPDB + VirusTotal + Shodan InternetDB |
| `Get All Threat Intelligence` | Dumps the complete local threat database |
| `Timeline Reconstruction` | Builds a chronological forensic timeline of suspicious events |
| `Execute Containment Playbook` | Generates network (iptables) and identity (usermod) containment commands |

## 🔒 Security Features

- **JWT Authentication** — HS256 signed tokens with configurable expiry, session persistence in MongoDB
- **bcrypt Password Hashing** — Secure password storage with salt rounds
- **Google OAuth 2.0** — ID token verification via Google's official auth library
- **Session Management** — Server-side session tracking with automatic expiry enforcement
- **CORS Configuration** — Restricted to known frontend origins
- **Auto-Response Dry Run** — Default safe mode logs containment commands without executing
- **Input Validation** — Pydantic models for all request/response schemas
- **Rate-Limited Threat Intel** — Respects API rate limits for AbuseIPDB and VirusTotal
- **Email Cooldown Tracking** — Per-IP cooldown prevents alert fatigue
